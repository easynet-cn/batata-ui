## 1. 软件概述

### 1.1 产品简介

Batata 微服务管理平台是一款基于 Rust 高性能后端与 Vue 3 现代前端构建的企业级微服务基础设施管理系统。本平台完全兼容 Apache Nacos 3.x 协议，同时提供 99% 的 HashiCorp Consul API 兼容性，可作为 Nacos 或 Consul 的高性能替代方案。

相比传统 Java 实现的 Nacos，Batata 具有以下显著优势：

- **极低内存占用**：运行时内存仅需 50-100MB，是 Java 版本的 1/10 至 1/20
- **极速启动**：1-2 秒即可完成启动，是 Java 版本的 1/30
- **双协议兼容**：同时兼容 Nacos 和 Consul 两大主流服务发现协议
- **AI 原生集成**：内置 MCP（Model Context Protocol）服务器注册与 A2A（Agent-to-Agent）智能体管理能力

### 1.2 核心功能

| 功能模块    | 说明                                                       |
| ----------- | ---------------------------------------------------------- |
| 配置管理    | 配置的创建、编辑、版本控制、灰度发布、导入导出、跨环境同步 |
| 服务管理    | 服务注册与发现、实例管理、健康检查、订阅者查询             |
| 命名空间    | 多环境配置隔离、租户容量管理                               |
| 集群管理    | 节点状态监控、Raft 共识协议、多数据中心同步                |
| 权限管理    | 用户管理、角色管理、细粒度权限控制（RBAC）                 |
| AI 管理     | MCP 服务器注册、A2A 智能体管理                             |
| Consul 兼容 | KV 存储、服务目录、ACL、Service Mesh                       |
| 可观测性    | Prometheus 指标、OpenTelemetry 链路追踪、审计日志          |
| 插件系统    | 可扩展的插件架构                                           |

### 1.3 适用场景

- 微服务架构下的服务注册与发现
- 分布式系统配置集中管理
- 多环境（开发、测试、预发布、生产）配置隔离
- 企业级权限控制与审计
- 云原生应用的服务治理
- AI/LLM 应用的工具与智能体管理

## 3. 安装与部署

### 3.1 单机模式部署

单机模式使用内嵌 RocksDB 存储，无需外部数据库，适合开发与测试环境。

```bash
# 单机模式（内嵌存储）- 推荐方式
./scripts/startup.sh -m standalone -p embedded

# 或使用旧版脚本
./scripts/start-embedded.sh
```

启动后访问控制台：`http://localhost:8081`

### 3.2 集群模式部署

集群模式需要外部数据库（MySQL 或 PostgreSQL），支持高可用和水平扩展。

```bash
# 配置数据库连接（conf/application.yml）
# db.url: mysql://user:password@localhost:3306/batata

# 配置集群成员（conf/cluster.conf）
# 192.168.1.1:8848
# 192.168.1.2:8848
# 192.168.1.3:8848

# 集群模式启动 - 推荐方式
./scripts/startup.sh

# 使用 MySQL 数据库
./scripts/startup.sh --db.url=mysql://user:pass@localhost:3306/batata

# 或使用旧版脚本
./scripts/start-server.sh
```

### 3.3 前后端分离部署

```bash
# 仅启动 Server（无控制台）- 推荐方式
./scripts/startup.sh -d server

# 仅启动 Console（连接远程 Server）
./scripts/startup.sh -d console

# Server + MCP Registry
./scripts/startup.sh -d serverWithMcp

# 或使用旧版脚本
./scripts/start-server.sh
./scripts/start-console.sh
```

### 3.4 Docker 部署

```bash
# 内嵌模式（无需数据库）
docker compose up batata

# 使用 MySQL
docker compose --profile mysql up

# 使用 PostgreSQL
docker compose --profile postgres up

# 完整监控栈（含 Prometheus + Grafana）
docker compose --profile mysql --profile monitoring up
```

**停止服务：**

```bash
# 使用脚本停止
./scripts/shutdown.sh

# 或使用 Docker Compose 停止
docker compose down
```

### 3.5 初始化管理员

首次部署后，需要初始化管理员密码：

```bash
./init-admin.sh
```

或通过浏览器访问管理员初始化页面完成设置。

## 6. 配置管理

配置管理是 Batata 的核心功能模块，提供配置的全生命周期管理。

### 6.1 配置列表

**路径：** 侧边栏 > 配置管理 > 配置列表

配置列表页面展示当前命名空间下的所有配置项。

**统计卡片：**
页面顶部展示四个统计卡片：配置总数、加密配置数、灰度配置数、分组数。

**操作功能：**

- **搜索筛选**：按 Data ID、Group、应用名称、标签进行搜索（支持模糊/精确模式）
- **高级搜索**：展开高级筛选条件，组合多个条件进行精确查询
- **分页浏览**：支持自定义每页显示数量
- **批量操作**：批量删除、批量导出选中的配置

### 6.2 创建配置

**路径：** 配置列表 > 创建配置

创建新配置需要填写以下信息：

| 字段     | 必填 | 说明                                               |
| -------- | ---- | -------------------------------------------------- |
| Data ID  | 是   | 配置的唯一标识符                                   |
| Group    | 是   | 配置分组，默认为 DEFAULT_GROUP                     |
| 配置格式 | 是   | 支持 TEXT、JSON、YAML、XML、HTML、Properties、TOML |
| 配置内容 | 是   | 使用代码编辑器编辑，支持语法高亮和自动补全         |
| 应用名称 | 否   | 关联的应用名称                                     |
| 标签     | 否   | 配置标签，用于分类和筛选                           |
| 描述     | 否   | 配置说明信息                                       |

代码编辑器基于 CodeMirror 6 构建，支持：

- 多语言语法高亮（JSON、YAML、XML、Properties 等）
- 代码折叠
- 行号显示
- 搜索替换
- 自动缩进

### 6.3 编辑配置

**路径：** 配置列表 > 点击操作列的"编辑"

进入配置编辑页面，可修改配置内容、格式、描述等信息。保存后自动记录历史版本。

### 6.4 配置详情

**路径：** 配置列表 > 点击 Data ID 链接

查看配置的完整信息，包括：

- 基本信息（Data ID、Group、格式、应用名称）
- 配置内容（只读模式展示）
- 元数据信息
- 操作按钮（编辑、删除）

### 6.5 版本历史

**路径：** 配置详情 > 历史版本

查看配置的所有历史变更记录：

- **版本列表**：展示每次变更的时间、操作人、操作类型
- **版本对比**：选择两个版本进行差异对比（Diff 视图）
- **版本回滚**：将配置回滚到指定的历史版本
- **查看详情**：查看某个历史版本的完整内容

### 6.6 灰度发布（Beta）

灰度发布允许将配置变更先推送给部分客户端进行验证，确认无误后再全量发布。

**灰度规则类型：**

- **IP 模式**：指定客户端 IP 列表，多个 IP 以逗号分隔
- **标签模式**：使用标签表达式匹配客户端（如 `env=gray AND region=cn-east`）

**操作流程：**

- **发布 Beta 配置**：在编辑页面选择灰度发布，设置灰度名称和规则
- **查看 Beta 状态**：在配置详情中查看当前灰度配置及规则
- **确认/取消灰度**：验证通过后确认全量发布，或取消灰度配置

### 6.7 配置加密

Batata 支持配置内容的自动加密与解密：

- **自动检测**：当 Data ID 以 `cipher-` 前缀开头时，系统自动启用加密
- **加密算法**：使用 AES-GCM 256 位加密
- **透明解密**：客户端通过 SDK 获取配置时自动解密

### 6.8 监听查询

**路径：** 侧边栏 > 配置管理 > 监听查询

查询当前正在监听配置变更的客户端：

- **按配置查询**：输入 Data ID 和 Group，查看监听该配置的所有客户端 IP
- **按 IP 查询**：输入客户端 IP，查看该客户端监听的所有配置

### 6.9 导入导出

**导出配置：**

- 在配置列表中勾选要导出的配置
- 点击"导出"按钮，下载 ZIP 格式的配置包

**导入配置：**

- 点击"导入"按钮，上传配置包文件
- 选择导入策略：
  - **跳过**：如果配置已存在则跳过
  - **覆盖**：如果配置已存在则覆盖
  - **终止**：如果配置已存在则中止导入

### 6.10 配置克隆

将选中的配置克隆到其他命名空间，适用于跨环境配置迁移。

### 6.11 配置同步

**路径：** 侧边栏 > 配置管理 > 配置同步

跨环境配置同步功能，支持将配置从当前环境同步到目标环境：

- **目标环境管理**：在 `application.yml` 中配置 `batata.sync.targets` 指定目标环境地址
- **同步策略**：
  - **跳过** (SKIP)：目标已存在的配置不覆盖
  - **覆盖** (OVERWRITE)：强制覆盖目标配置
  - **中止** (ABORT)：遇到冲突时中止同步
- **同步历史**：查看历史同步记录和结果

## 7. 服务管理

服务管理模块提供微服务的注册、发现和实例管理功能。

### 7.1 服务列表

**路径：** 侧边栏 > 服务管理 > 服务列表

展示当前命名空间下的所有注册服务：

| 列名       | 说明                |
| ---------- | ------------------- |
| 服务名称   | 服务的唯一名称      |

展示服务的详细信息：

| 列名      | 说明             |

**实例操作：**

- 编辑集群健康检查配置

### 7.3 订阅者列表

**路径：** 侧边栏 > 服务管理 > 订阅者列表

查看正在订阅服务变更通知的客户端：

- 客户端 IP 地址
- 订阅的服务名称
- 应用标识

## 9. 集群管理

**路径：** 侧边栏 > 集群管理

### 9.1 节点列表

展示集群中所有节点的状态信息：

| 列名     | 说明                                                       |
| -------- | ---------------------------------------------------------- |
| 节点地址 | 节点的 IP 地址和端口                                       |
| 状态     | UP（在线）/ DOWN（离线）/ SUSPICIOUS（疑似故障）           |
| 节点能力 | 支持的功能列表（Remote 远程连接、Metrics 指标、Raft 共识） |

- **AP 模式（Distro）**：最终一致性，适用于服务发现等对可用性要求高的场景

## 10. 权限管理

Batata 提供基于角色的访问控制（RBAC）体系。

### 10.1 用户管理

**路径：** 侧边栏 > 权限管理 > 用户列表

- **创建用户**：设置用户名和密码
- **修改密码**：重置用户密码
- **删除用户**：删除用户账户
- **搜索用户**：按用户名搜索

### 10.2 角色管理

**路径：** 侧边栏 > 权限管理 > 角色列表

- **绑定角色**：将角色分配给用户
- **解绑角色**：移除用户的角色绑定
- **搜索**：按角色名或用户名搜索

### 10.3 权限管理

**路径：** 侧边栏 > 权限管理 > 权限列表

配置细粒度的资源访问权限：

| 字段 | 说明                       |
| ---- | -------------------------- |
| 角色 | 权限所属角色               |
| 资源 | 资源标识（支持通配符 `*`） |
| 动作 | 读取（r）/ 写入（w）       |

**权限格式：** `{命名空间}:{分组}:{资源类型}/{资源名称}`

例如：`public:DEFAULT_GROUP:config/application.yml:r` 表示对 public 命名空间下 DEFAULT_GROUP 分组的 application.yml 配置具有读取权限。

### 10.4 认证方式

Batata 支持多种认证方式：

- **本地认证**：用户名密码（JWT Token）
- **LDAP**：企业级目录服务集成
- **OAuth2/OIDC**：支持 Google、GitHub、Microsoft 等第三方登录

## 11. AI 智能管理

Batata 内置 AI 原生管理能力，支持 MCP 和 A2A 协议。

### 11.1 MCP 服务器管理

**路径：** 侧边栏 > AI 管理 > MCP 服务器

MCP（Model Context Protocol）是 AI/LLM 应用的工具注册协议。

**统计卡片：**
页面顶部展示四个统计卡片：MCP 服务器总数、已启用数、工具总数、服务器类型数。

**MCP 服务器列表：**

- 展示所有注册的 MCP 服务器
- 显示服务器类型（stdio、SSE、HTTP）
- 显示启用/禁用状态
- 显示可用工具数量

**创建 MCP 服务器：**

| 字段       | 说明                     |
| ---------- | ------------------------ |
| 服务器名称 | MCP 服务器名称           |
| 连接类型   | stdio / SSE / HTTP       |
| 连接配置   | 命令行参数或 URL         |
| 环境变量   | 服务器运行所需的环境变量 |
| 工具列表   | 服务器提供的工具定义     |

**MCP 服务器详情：**

- 查看服务器元数据
- 浏览可用工具列表
- 查看工具的输入/输出 Schema

**导入功能：**

- 从现有 MCP 服务器导入工具定义
- 从 MCP Registry（注册表）搜索并导入服务器

## 15. 链路追踪

**路径：** 侧边栏 > 链路追踪

提供分布式链路追踪集成能力。Batata 通过 OpenTelemetry (OTLP) 协议导出追踪数据到外部追踪系统。

### 15.1 配置追踪

在 `application.yml` 中启用 OpenTelemetry：

```yaml
batata:
  otel:
    enabled: true
    exporter:
      otlp:
        endpoint: http://jaeger:4317
    service:
      name: batata-server
```

### 15.2 追踪查询

当 OTLP 已配置时，页面显示追踪后端信息和连接状态。完整追踪数据请查询 Jaeger/Tempo 等追踪后端。

## 19. 常见问题

### Q1: 忘记管理员密码怎么办？

可通过命令行工具重置管理员密码：

```bash
./init-admin.sh
```

### Q2: 如何从 Nacos 迁移到 Batata？

Batata 完全兼容 Nacos V2/V3 API，现有的 Nacos SDK 客户端无需修改代码，只需将服务端地址指向 Batata 即可。配置数据可通过导出/导入功能迁移。

### Q3: 如何从 Consul 迁移到 Batata？

Batata 提供 99% 的 Consul API 兼容性。现有 Consul 客户端只需修改服务端地址即可接入。KV 数据可通过 JSON 格式导入导出。

### Q4: 集群模式下节点数量有什么要求？

建议使用 3 个或 5 个节点组成集群（奇数节点），以满足 Raft 协议的多数派要求。

### Q5: 如何配置 HTTPS？

可在 Batata 前部署 Nginx 或其他反向代理来提供 HTTPS 支持。

### Q6: 支持哪些配置格式？

支持 TEXT、JSON、YAML、XML、HTML、Properties、TOML 共 7 种配置格式，编辑器提供对应的语法高亮支持。

### Q7: 如何开启鉴权？

在配置文件 `conf/application.yml` 中设置：

```yaml
nacos:
  core:
    auth:
      enabled: true
```
